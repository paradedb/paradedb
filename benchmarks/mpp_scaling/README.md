# MPP linear-scaling micro-benchmark

Measures whether `paradedb.enable_mpp = on` scales linearly with `paradedb.mpp_worker_count`.

## What "linear" means here

For N workers, runtime should be ≈ baseline/N. Efficiency = (baseline/runtime) / N. We aim for ≥0.7 (70% of ideal) at N=2/4/8 on the per-query baseline.

Baseline is **MPP off** with `max_parallel_workers_per_gather=0` (true serial). Not "PG native parallel scan" — that's a different code path.

## Query shapes

Each chosen for being theoretically embarrassingly parallel (work splits cleanly across workers, gather is small).

| ID  | Shape                                                                                                    | Why it should scale                                |
| --- | -------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| Q1  | `SELECT COUNT(*) FROM t WHERE body @@@ 'term'`                                                           | Pure scan, no shuffle, leader gather is single row |
| Q2  | `SELECT category, COUNT(*) FROM t WHERE body @@@ 'term' GROUP BY category` (low card, ~10 groups)        | Parallel partial agg + tiny final (10 rows)        |
| Q3  | `SELECT user_id, COUNT(*) FROM t WHERE body @@@ 'term' GROUP BY user_id` (high card, ~10% rows distinct) | Parallel partial + shuffle + parallel final        |
| Q4  | `SELECT * FROM t WHERE body @@@ 'term' ORDER BY score LIMIT 10`                                          | Parallel TopN + leader merge of 10×N rows          |

Q1/Q2 should be the easiest to make linear. Q3 is the known hard case (shuffle bandwidth). Q4 has serial leader merge.

## Data

Synthetic table generated by `setup.sql`:

- `id BIGINT PRIMARY KEY`
- `category INT` (10 distinct values, low-card)
- `user_id BIGINT` (high-card, N_ROWS/10 distinct)
- `score INT` (random 0..1000)
- `body TEXT` (BM25-indexed; contains "term" on ~30% of rows)

Sizes tested: 1M (fast iter), 10M (real signal), 25M (stress).

## Worker counts

N = 1 baseline (MPP off), then MPP on at 2, 4, 8 producer workers (so `mpp_worker_count = 3, 5, 9` — value includes the leader).

## Per-run protocol

1. `DISCARD ALL`
2. `SELECT pg_prewarm(...)` to keep buffer cache warm
3. `EXPLAIN (ANALYZE, FORMAT JSON) <query>` ×3 (median wall time)
4. Capture: wall_ms, peak memory, MPP stage shapes from EXPLAIN

## Iteration loop

1. Measure baseline + N=2/4/8.
2. If any query's efficiency < 0.7 at N=4, dig:
   - Per-fragment timing (mpp_log)
   - shm_mq backpressure (`grep send_wait`)
   - Hash join build serial cost (EXPLAIN says so)
3. Apply fix on a new commit on top of #5136.
4. Re-measure.
